  <action>
    Apply these MECHANICAL rules uniformly across every f-string log call site. NO
    business-logic changes; only the log statement rewrites.

    Rule A — informational logs:
        logger.info(f"Step 1: Stored {count} HOSE stock listings")
        becomes
        logger.info("pipeline.listings.stored", step=1, count=count)

    Rule B — exception logs INSIDE except blocks:
        try:
            ...
        except Exception as e:
            logger.error(f"Telegram send failed: {e}")
        becomes
        try:
            ...
        except Exception:
            logger.exception("telegram.send.failed")
        (logger.exception captures stack via loguru exc_info; do NOT pass the exception
        instance as a kwarg.)

    Rule C — warning logs with variable interpolation:
        logger.warning(f"Skipping {symbol}: {e}")
        becomes
        logger.warning("crawl.symbol.skipped", symbol=symbol, error=str(e))

    Rule D — Loguru positional placeholder form is also acceptable when the message is
    inherently a sentence (preserve readability):
        logger.info("score computed for {symbol} = {score}", symbol=symbol, score=score)
        — equivalent to kwargs form for loguru's serialize=True; the structured extras
        come from the kwargs.

    Event-name conventions (ASCII, dot-separated, lowercase):
      - crawlers: crawl.<entity>.<action>           (e.g. crawl.prices.fetched)
      - repositories: <entity>_repo.<action>        (e.g. price_repo.bulk_inserted)
      - services: <area>.<action>                   (e.g. analysis.score.computed)
      - notifications: telegram.send.<status>
      - ai: ai.client.<action>
    Vietnamese is OK in trailing message text but event keys MUST be ASCII for grep-ability.

    Per-file workflow:
      1. Read the file
      2. For each `logger.X(f"...")` call, apply rules A/B/C/D
      3. Choose an event name that maps cleanly to the call site's intent
      4. Move every interpolated variable into a kwarg
      5. Save and run the lint script — must exit 0 once ALL files are done

    Do NOT touch:
      - api/app.py (already clean from 22-03)
      - scheduler/scheduler.py (already clean from 22-04)
      - services/pipeline.py (already clean from 22-04)
      - any test file (tests will be re-checked separately)

    If a call site is in critical hot-path code (e.g., per-row logging in a tight loop),
    DO NOT introduce extra string formatting work — kwargs form is identical-or-cheaper
    than f-string interpolation.

    For genuinely awkward cases (e.g., dynamically-generated message text), the escape
    hatch `# noqa: log-fstring` is allowed. Use it only when you cannot reasonably
    convert (target: < 5 uses across the entire phase). Document each use in the
    SUMMARY.md.
  </action>
