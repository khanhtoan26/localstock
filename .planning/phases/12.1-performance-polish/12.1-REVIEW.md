## Warnings

### WR-01: focusedJobId not cleared when target row is absent from DOM

**File:** `apps/helios/src/components/admin/job-monitor.tsx:90-101`
**Issue:** When `focusedJobId` is set (via toast "View job" action) but the target row is not present in `rowRefs.current` — e.g., hidden by an active status or type filter — the `useEffect` silently does nothing. `onFocusHandled()` is never called, so `focusedJobId` remains set in the parent. Because `focusedJobId` is in the effect's dependency array and doesn't change, the effect won't re-fire when the user later changes filters to reveal the row. The result: the highlight is missed entirely, and the parent holds a stale `focusedJobId` until the next job transition overwrites it.

While the default filters are "all" (making this edge case unlikely in the normal toast-action flow), it's a logic gap that can confuse users who have customized their filter view.

**Fix:** Add an `else` branch that resets the focused state when the row can't be found, and optionally reset filters to "all" when navigating from a toast action:

```typescript
useEffect(() => {
  if (focusedJobId == null) return;
  const row = rowRefs.current.get(focusedJobId);
  if (row) {
    row.scrollIntoView({ behavior: "smooth", block: "nearest" });
    row.classList.add("job-row-highlight");
    const timer = setTimeout(() => {
      row.classList.remove("job-row-highlight");
      onFocusHandled?.();
    }, 2300);
    return () => clearTimeout(timer);
  } else {
    // Row not in DOM (filtered out or not yet rendered) — reset to avoid stale state
    onFocusHandled?.();
  }
}, [focusedJobId, onFocusHandled]);
```

Alternatively, reset `statusFilter` and `typeFilter` to `"all"` when `focusedJobId` changes to ensure the target row is always visible.

## Info

### IN-01: Cross-file magic number coupling for animation duration

**File:** `apps/helios/src/components/admin/job-monitor.tsx:99`
**Issue:** The `2300` timeout value on line 99 must exactly match the `job-highlight` animation duration in `globals.css:170`. The inline comment documents this, but the coupling is implicit — changing one without the other would cause the highlight class to be removed before/after the animation ends.
**Fix:** Extract into a shared constant, or add a `data-*` attribute / CSS custom property that JS can read. For a lightweight approach, define the constant in a shared location:

```typescript
// In a shared constants file or at the top of job-monitor.tsx
const JOB_HIGHLIGHT_DURATION_MS = 2300;
```

### IN-02: No exhaustive check in invalidateForJob switch

**File:** `apps/helios/src/lib/queries.ts:259-300`
**Issue:** The switch on `job.job_type` covers all 5 current union members but lacks a `default` case or TypeScript exhaustive `never` assertion. If a new job type is added to the `AdminJob` type in the future, the switch will silently do nothing for it.
**Fix:** Add an exhaustive check after the last case:

```typescript
    case "pipeline":
      // ... existing code ...
      break;

    default: {
      const _exhaustive: never = job.job_type;
      void _exhaustive;
    }
```

### IN-03: onTransition callback stability requirement not documented at call site

**File:** `apps/helios/src/hooks/use-job-transitions.ts:19-20`
**Issue:** The `onTransition` parameter is included in the `useEffect` dependency array (line 59), so callers must wrap it in `useCallback` to avoid unnecessary effect re-runs on every render. The plan documents this requirement (Pitfall 1), and the current caller in `page.tsx` correctly uses `useCallback`, but the hook's JSDoc doesn't mention this contract.
**Fix:** Add a note to the JSDoc:

```typescript
/**
 * ...existing docs...
 *
 * @param onTransition - Called for each detected transition. **Must be wrapped
 *   in useCallback** by the caller to prevent unnecessary effect re-runs.
 */
```
