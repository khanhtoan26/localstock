## Summary

Phase 12.1 is a behavioral polish phase for the existing `/admin` console delivered in Phase 12. The three core workstreams are: (1) targeted cache invalidation when admin-triggered jobs complete, (2) toast feedback with row-highlight + auto-focus UX for job status transitions, and (3) optional performance tuning of the stock table. No new pages, components, or layouts are introduced.

The existing codebase is well-structured for these changes. The `useAdminJobs` hook already polls every 3 seconds when active jobs exist, providing a natural detection point for job status transitions. The `params.symbols` field on each `AdminJob` record gives exactly the information needed to target cache invalidation. Sonner 2.0.7 (already installed) supports `action` buttons on toasts natively. CSS keyframe animations using `color-mix()` are already used in the `StatusBadge` component, confirming browser/build compatibility.

**Primary recommendation:** Implement a job-transition detection layer (via `useRef` tracking previous job statuses against current poll data) that triggers both targeted `queryClient.invalidateQueries()` calls and toast notifications with action buttons — all coordinated through the existing `AdminPage` component that already manages tab state.

### Pattern 1: Job Transition Detection via useRef Diff

**What:** Track previous poll data in a `useRef`, compare on each re-render to detect status transitions.
**When to use:** When you need to detect state changes in polled data without server-sent events.

```typescript
// Source: React docs useRef + TanStack Query polling pattern [VERIFIED: types.ts AdminJob interface]
import { useRef, useEffect, useCallback } from "react";
import { useQueryClient } from "@tanstack/react-query";
import { useAdminJobs } from "@/lib/queries";
import type { AdminJob } from "@/lib/types";

interface JobTransition {
  job: AdminJob;
  previousStatus: string;
}

export function useJobTransitions(
  onTransition: (transition: JobTransition) => void
) {
  const { data } = useAdminJobs();
  const prevJobsRef = useRef<Map<number, string>>(new Map());

  useEffect(() => {
    if (!data?.jobs) return;

    const transitions: JobTransition[] = [];
    const currentMap = new Map<number, string>();

    for (const job of data.jobs) {
      currentMap.set(job.id, job.status);
      const prevStatus = prevJobsRef.current.get(job.id);

      // Detect transition: was running/pending → now completed/failed
      if (
        prevStatus &&
        prevStatus !== job.status &&
        (job.status === "completed" || job.status === "failed")
      ) {
        transitions.push({ job, previousStatus: prevStatus });
      }
    }

    prevJobsRef.current = currentMap;

    for (const t of transitions) {
      onTransition(t);
    }
  }, [data, onTransition]);
}
```

### Pattern 2: Targeted Cache Invalidation by Job Type + Symbols

**What:** Map each job type to the exact query keys that need invalidation, using symbols from `job.params.symbols`.
**When to use:** After detecting a job completion, invalidate only the caches that the job's output would have changed.

```typescript
// Source: TanStack Query v5 invalidateQueries [VERIFIED: queryClient.d.ts]
// Key insight: queryKey prefix matching — ["scores"] matches ["scores", "VNM"]
// and ["scores", "top", 20] — so invalidating ["scores"] refreshes ALL score queries.

function invalidateForJob(
  queryClient: QueryClient,
  job: AdminJob
) {
  const symbols = (job.params?.symbols as string[]) ?? [];

  switch (job.job_type) {
    case "crawl":
      // Crawl updates price data
      for (const s of symbols) {
        queryClient.invalidateQueries({ queryKey: ["prices", s] });
      }
      queryClient.invalidateQueries({ queryKey: ["admin", "stocks"] });
      break;

    case "analyze":
      // Analysis updates technical + fundamental data
      for (const s of symbols) {
        queryClient.invalidateQueries({ queryKey: ["technical", s] });
        queryClient.invalidateQueries({ queryKey: ["fundamental", s] });
      }
      break;

    case "score":
      // Scoring updates per-symbol scores + top scores ranking
      for (const s of symbols) {
        queryClient.invalidateQueries({ queryKey: ["scores", s] });
      }
      queryClient.invalidateQueries({ queryKey: ["scores", "top"] });
      break;

    case "report":
      // Report updates per-symbol report
      for (const s of symbols) {
        queryClient.invalidateQueries({ queryKey: ["report", s] });
      }
      break;

    case "pipeline":
      // Full pipeline: invalidate everything for affected symbols
      for (const s of symbols) {
        queryClient.invalidateQueries({ queryKey: ["prices", s] });
        queryClient.invalidateQueries({ queryKey: ["technical", s] });
        queryClient.invalidateQueries({ queryKey: ["fundamental", s] });
        queryClient.invalidateQueries({ queryKey: ["scores", s] });
        queryClient.invalidateQueries({ queryKey: ["report", s] });
      }
      queryClient.invalidateQueries({ queryKey: ["scores", "top"] });
      queryClient.invalidateQueries({ queryKey: ["admin", "stocks"] });
      break;
  }
}
```

**TanStack Query key matching behavior** (verified from source): `invalidateQueries({ queryKey: ["scores", "VNM"] })` uses `partialMatchKey` — it matches any query whose key starts with `["scores", "VNM"]`. So `["scores", "VNM"]` as a filter matches the exact query `["scores", "VNM"]` but NOT `["scores", "top", 20]`. This is exactly what we need for symbol-scoped invalidation. [VERIFIED: partialMatchKey implementation in query-core/utils.js]

### Anti-Patterns to Avoid

- **Broad invalidation after every mutation:** Don't `queryClient.invalidateQueries()` with no filter — this refetches everything. Use targeted keys per D-01.
- **Invalidation on trigger (not completion):** Don't invalidate caches when the trigger request succeeds — data hasn't changed yet. Wait for job completion per D-02.
- **Adding new polling hooks:** Don't create a second polling mechanism. The existing `useAdminJobs` with `refetchInterval: 3000` is the single source of truth for job status per D-03.
- **Using `useEffect` with missing deps for transition detection:** The `onTransition` callback passed to `useJobTransitions` must be wrapped in `useCallback` to prevent infinite re-render loops.
- **Direct DOM manipulation for highlight:** Don't bypass React's ref system. Use callback refs on `<TableRow>` elements to build a ref map, then access via the map.

## Common Pitfalls

### Pitfall 1: Stale Closure in Transition Callback
**What goes wrong:** The `onTransition` callback captures stale tab state or stale `queryClient` reference, causing invalidation to fire with outdated context or toast actions to switch to wrong tab.
**Why it happens:** React hooks close over the render-time values. If `onTransition` isn't memoized with `useCallback`, it re-creates every render, potentially causing the `useEffect` in the transition hook to fire with stale refs.
**How to avoid:** Wrap `onTransition` in `useCallback` with correct dependency array. Use `useQueryClient()` inside the callback (it returns a stable reference). For tab switching, pass `setActiveTab` directly (React state setters are stable).
**Warning signs:** Toast action buttons don't switch tabs correctly; invalidation seems to not trigger.

### Pitfall 2: Transition Detection on Initial Load
**What goes wrong:** On first mount, `prevJobsRef` is empty, so every existing completed/failed job triggers a "transition" toast, flooding the user with historical notifications.
**Why it happens:** The diff logic treats "not seen before" as a transition.
**How to avoid:** Skip transition detection on the first data load. Use a boolean ref (`isInitialLoad`) that's set to `false` after the first `useEffect` run. Only start detecting transitions from the second poll onwards.
**Warning signs:** Multiple toasts appear immediately on page load for old jobs.

### Pitfall 3: Race Between Tab Switch and scrollIntoView
**What goes wrong:** The toast action button sets `activeTab("jobs")` and `focusedJobId` simultaneously, but the `JobMonitor` component hasn't mounted yet (it was in an inactive tab), so the row ref doesn't exist and `scrollIntoView` silently fails.
**Why it happens:** Tab content may be unmounted when the tab is inactive (depends on `TabsContent` behavior — some implementations use `forceMount`, others lazy-mount).
**How to avoid:** Option A: Use `forceMount` on `TabsContent value="jobs"` so JobMonitor is always in DOM (hidden via CSS). Option B: In the `useEffect` that handles `focusedJobId`, add a short delay (`requestAnimationFrame` or `setTimeout(0)`) to wait for the tab to render. Option A is simpler and more reliable.
**Warning signs:** Highlight works when you're already on the Jobs tab, but fails when coming from another tab via toast action.

### Pitfall 4: Multiple Rapid Transitions
**What goes wrong:** If a batch of jobs all complete on the same poll cycle, the transition handler fires multiple times synchronously, potentially causing React batching issues or toast spam.
**Why it happens:** When 5 symbols are crawled together, the batch job completes as one job with `params.symbols: ["VNM", "FPT", "HPG", "SSI", "VCB"]`. But if the user triggered multiple separate operations, multiple jobs can transition simultaneously.
**How to avoid:** The transition detection loop should collect all transitions first, then process them. For invalidation, TanStack Query's `notifyManager.batch()` handles this internally. For toasts, showing one per job is fine — Sonner stacks them naturally.
**Warning signs:** Duplicate toasts, or only the last transition's invalidation taking effect.

### Pitfall 5: AdminJob.params Type Safety
**What goes wrong:** Accessing `job.params?.symbols` without type guard could fail if params structure changes or is null.
**Why it happens:** `params` is typed as `Record<string, unknown> | null` — no type safety on the `symbols` field.
**How to avoid:** Add a helper function: `function getJobSymbols(job: AdminJob): string[] { return Array.isArray(job.params?.symbols) ? job.params.symbols as string[] : []; }`. Use this consistently.
**Warning signs:** Runtime errors when accessing params on jobs created by pipeline or other trigger types.

### Pitfall 6: CSS `color-mix()` Compatibility
**What goes wrong:** `color-mix()` might not be supported in very old browsers.
**Why it happens:** `color-mix()` is relatively new CSS (2023).
**How to avoid:** Not a real concern for this project — `color-mix()` is already used in `StatusBadge` (line 9-11 of status-badge.tsx), confirming the build/runtime supports it. All modern browsers (Chrome 111+, Firefox 113+, Safari 16.2+) support it. [VERIFIED: status-badge.tsx already uses color-mix in production]
**Warning signs:** None expected — already shipping.

### Complete Toast Notification on Job Completion

```typescript
// Source: sonner 2.0.7 types [VERIFIED: sonner/dist/index.d.ts]
// Combined with next-intl pattern [VERIFIED: existing toast usage in stock-table.tsx]

import { toast } from "sonner";
import { useTranslations } from "next-intl";

const t = useTranslations("admin");

function showJobToast(
  job: AdminJob,
  setActiveTab: (tab: string) => void,
  setFocusedJobId: (id: number) => void,
) {
  const symbols = getJobSymbols(job);
  const symbolStr = symbols.length > 0 ? symbols.slice(0, 3).join(", ") : null;
  const hasMore = symbols.length > 3 ? ` +${symbols.length - 3}` : "";

  const message = symbols.length > 0
    ? t("toast.jobCompleted", { type: job.job_type, symbols: symbolStr + hasMore })
    : t("toast.jobCompletedGeneric", { type: job.job_type });

  const method = job.status === "completed" ? toast.success : toast.error;
  const duration = job.status === "completed" ? 5000 : 8000;

  method(message, {
    duration,
    action: {
      label: t("toast.viewJob"),
      onClick: () => {
        setActiveTab("jobs");
        setFocusedJobId(job.id);
      },
    },
  });
}
```

## Open Questions

1. **TabsContent mounting behavior**
   - What we know: The app uses `@base-ui/react` Tabs, not shadcn's Radix-based tabs. The existing code uses `Tabs`, `TabsList`, `TabsTrigger`, `TabsContent` imports.
   - What's unclear: Whether inactive `TabsContent` children are unmounted or hidden. This affects whether `forceMount` is needed for the scroll-into-view pattern.
   - Recommendation: Test empirically during implementation. If refs are null when switching tabs, add `forceMount` or a `requestAnimationFrame` delay.

2. **Performance of ~400 symbols in invalidation loop**
   - What we know: The worst case is a full pipeline trigger on all ~400 tracked stocks, which would call `invalidateQueries` ~2400 times (6 key types × 400 symbols).
   - What's unclear: Whether TanStack Query's batching (`notifyManager.batch`) handles this efficiently.
   - Recommendation: In practice, admin triggers are almost always for a small selection (5-20 symbols). For all-stocks pipeline, consider invalidating by prefix without symbol specificity: `queryClient.invalidateQueries({ queryKey: ["prices"] })` etc. — one call per data type instead of per symbol.
