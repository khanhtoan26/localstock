# Phase 12.1: Performance & Polish - Context

**Gathered:** 2026-04-23
**Status:** Ready for planning

<domain>
## Phase Boundary

Polish the existing `/admin` experience delivered in Phase 12. Focus on correctness and perceived quality of the current admin console: cache freshness after admin-triggered jobs, clearer job feedback, and performance improvements that fit the existing architecture. This phase does not introduce new report-specific UI; Phase 13 owns new AI report generation interactions and preview flows.

</domain>

<decisions>
## Implementation Decisions

### Cache freshness after admin actions
- **D-01:** Use targeted query invalidation based on the affected symbol(s) and job type instead of broad refetching of all stock-related data after every admin action.
- **D-02:** Trigger invalidation when an admin job transitions to `completed` or `failed` through the existing jobs polling flow, not immediately when the trigger request is sent.

### Job feedback UX
- **D-03:** Keep the Jobs tab as the single source of truth for background job status; do not add a separate cross-page command-center surface in this phase.
- **D-04:** Add toast feedback plus auto-focus/bring attention to the newly created or just-finished job in the Jobs tab.

### Phase boundary with Phase 13
- **D-05:** Phase 12.1 covers the polish and correctness changes discussed here for the current admin console only.
- **D-06:** New report-specific UI remains out of scope for 12.1 and stays in Phase 13.

### the agent's Discretion
- Performance optimization approach for the stock table flows may be chosen during planning and implementation, as long as it stays within the current admin-console architecture and is appropriate for the existing HOSE-scale dataset.

</decisions>

<specifics>
## Specific Ideas

- Keep Jobs tab as the operational center for long-running work.
- When a background job completes, the admin UI should make the result obvious without forcing the user to hunt for the updated row.
- Avoid premature heavy solutions unless the current ~400-symbol workload justifies them.

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `useTrackedStocks`, `useAdminJobs`, `useAdminJobDetail`, and admin trigger mutations in `apps/helios/src/lib/queries.ts` — existing query/mutation layer to extend for smarter invalidation and focus behavior
- `Table`, `Button`, `Input`, `Checkbox`, `Badge`, `EmptyState`, `ErrorState`, and `Skeleton` components — established admin UI primitives already used in the current page
- `sonner` toast usage in `stock-table.tsx` and `pipeline-control.tsx` — existing notification mechanism to reuse for completion/failure feedback

### Established Patterns
- Admin data loading uses TanStack Query with explicit query keys and mutation-driven invalidation
- Large stock lists are currently handled client-side with memoized filter/sort logic and 50-row pagination
- Background work visibility is driven by polling in the Jobs tab rather than SSE or websocket updates

### Integration Points
- `apps/helios/src/app/admin/page.tsx` — place to coordinate tab focus when a job is created or completes
- `apps/helios/src/components/admin/job-monitor.tsx` — place to highlight the relevant row when a job becomes actionable
- `apps/helios/src/lib/queries.ts` — place to map job completion events to the exact caches that need invalidation, including admin queries and symbol-scoped report/score queries
